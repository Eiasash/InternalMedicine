## Deliberately left (with rationale — needs Eias's decision where noted)

### 1. `shared/install-promo.js` L82 — `no-empty` error (NOT FIXED — needs Eias)
The 3rd eslint error: `catch {}` empty block on L82. **This is a cross-repo drift-tracked shared file** — md5 `4ef4dafe83fb9a1828761fa7248ed521`, byte-identical with `.shared/install-promo.js` (workspace canonical), Geriatrics, and FamilyMedicine. Fixing it here alone would create the exact cross-repo drift `auto-audit` warns on within 30 min, and is a single-repo change to a multi-repo contract.
**Root cause is the lint config, not the code:** `eslint.config.js` sets `'no-empty': ['error', { allowEmptyCatch: true }]` for `src/**/*.js` but the `shared/**/*.js` block (lines 70-82) does **not** carry that override, so it inherits the stricter recommended default. The right fix is in the rule (add `allowEmptyCatch` to the shared block, or `-Confirm` the empty catch is intentional), and should be made on the canonical `.shared/` source + propagated to all siblings in one cross-repo PR — not here. **Decision for Eias:** approve a workspace-level cross-repo PR to either (a) align the `shared/**` eslint block with the `src/**` `allowEmptyCatch` setting, or (b) add a comment to the empty catch. Same treatment as `fsrs.js`.

### 2. 77 residual `no-unused-vars` warnings — left intentionally
Breakdown: ~55 `catch(e)` where `e` is unused, ~7 unused function args, ~6 unused array-destructure elements, plus a few in-function scaffolding vars. These are **cosmetic style noise, not dead code or bugs**, and the eslint config marks them `warn` (non-blocking; CI `ci.yml`/`integrity-guard.yml` do not gate on eslint). Sweeping 55+ `catch(e)→catch`/`_e` renames is high-churn across many files and violates the repo's "touch only what you must / minimum code" working rules for zero functional benefit. **Note:** the eslint config lacks `caughtErrorsIgnorePattern` — adding it (config change, out of scope here) would silence all 55 at once and is the right long-term fix; flagged for a future config-tuning pass.

### 3. `dir="rtl"` in `src/features/study_plan/index.js` (4 sites) + `src/ui/quiz-view.js:189` — left, NOT a bug
Workspace CLAUDE.md prefers `dir="auto"` + `unicode-bidi:plaintext` over forced `dir="rtl"`, and constants.js v10.4.2 documents a conversion campaign. **However:** the study_plan blocks (L148/200/214/247) are Hebrew-primary cards that force RTL deliberately and **already isolate every interpolated English fragment** at finer grain (`dir="${heDir(...)}"`, `direction:ltr;display:inline-block` spans, `direction:ltr;text-align:left` keyword rows). This is structurally different from the v10.4.2 campaign target (interpolated error/eFlag text with no inner isolation). `quiz-view.js:189` is a fully static loading message with zero interpolation. Converting these to `dir="auto"` is a behavioral/visual change (auto guesses from first strong char — risky for emoji/English-leading cards), out of bug-fix scope, and unverifiable without a browser. **Not a bug; deferred as cosmetic. Recommend a future browser-eyeballed bidi-hygiene pass if convention alignment is desired.**
